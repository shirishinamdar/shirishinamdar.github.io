Ophcrack feels like a tool that shouldn't work in 2026. You boot a Linux LiveCD off a USB stick, point it at a Windows machine's hard drive, and a few minutes later it prints the user's password. No exploit, no malware, no clever cryptographic break, just a precomputed lookup table and an old hash format that doesn't fight back.

It works because Windows protects its password file by *running*, not by encrypting it. Boot something else and the protection evaporates. This post is my walkthrough of that idea, end to end, against a personal Dell Precision M70 running Windows XP, the original target Ophcrack was built for.

> Password cracking against systems you do not own or have explicit written authorization to test is unlawful in most jurisdictions, including the United States (Computer Fraud and Abuse Act), the United Kingdom (Computer Misuse Act 1990), and India (Information Technology Act, Section 66).

---

## Background: Why This Attack Works

Windows does not store user passwords in plaintext. Instead, it stores one-way hash representations of passwords inside the **SAM** registry hive at `C:\WINDOWS\system32\config\SAM`. Two hash formats are historically relevant:

| Hash Type | Algorithm | Salting | Notable Weakness |
| --- | --- | --- | --- |
| **LM** (LAN Manager) | DES-based | None | Password uppercased and split into two 7-character halves, each hashed independently |
| **NTLM** | MD4 over UTF-16 | None | No salt; vulnerable to precomputed lookups |

**Rainbow tables** are a time-memory trade-off introduced by Philippe Oechslin in 2003. Rather than precomputing every possible hash (storage prohibitive) or brute-forcing every candidate (compute prohibitive), a rainbow table stores chains generated by alternating hashing and reduction functions. Recovering a password reduces to walking a chain, a small, bounded operation against a multi-gigabyte table. The technique works against unsalted hash schemes such as LM and raw NTLM. It does not work against modern salted, computationally expensive schemes such as bcrypt or Argon2.

While Windows holds an active session, the SAM hive is locked and cannot be read by user-space tools. The standard offline bypass is to boot a separate operating system from removable media, mount the host disk, and read the SAM and SYSTEM hives directly. Ophcrack's LiveCD is built around this workflow.

## Project Setup

| Item | Detail |
| --- | --- |
| Target hardware | Dell Precision M70 (personal project machine) |
| Target operating system | Windows XP SP3 |
| Attack tool | Ophcrack LiveCD (Windows XP / 7 build), booted from USB |
| Rainbow tables | Ophcrack Free Small (alphanumeric, ~700 MB) |
| Test account | `IETNOVA`, limited (non-administrator) permissions |
| Test password | `computer` (eight lowercase letters, dictionary word) |

A weak, lowercase, dictionary-word password was selected intentionally to demonstrate the baseline case. Stronger passwords would either require larger paid tables or fall outside Ophcrack's table coverage entirely.

## 1. Creating the Test Account

A local user account named `IETNOVA` was created in **Computer Management** with limited (non-administrator) permissions, and assigned the password `computer`.

---

## 2. Booting Ophcrack from USB

The system was rebooted, and the BIOS boot menu was used to select the USB storage device. The point of this step is not the boot menu itself but the underlying threat model: any party who controls what a machine boots also controls which software can read its disk. This is the same threat model that applies to a stolen laptop without full-disk encryption.

![Selecting the USB Storage device in BIOS to boot Ophcrack instead of Windows.](/assets/img/blog/ophcrack-rainbow-tables/image2.jpeg)

---

## 3. SAM Extraction and Table Loading

The Ophcrack LiveCD boots into a minimal Linux environment, automatically mounts the Windows partition, and locates both the SAM and SYSTEM registry hives. The SYSTEM hive is required because Windows obfuscates the SAM hashes with a key (SYSKEY) derived from values inside SYSTEM. Once both hives are read, Ophcrack extracts the LM and NTLM hashes for each local account and prepares them for comparison against the loaded rainbow tables.

![Ophcrack initial screen showing automatic SAM extraction in progress.](/assets/img/blog/ophcrack-rainbow-tables/image3.jpeg)

---

## 4. Hash Recovery

After several minutes, Ophcrack completed the table walk and printed the recovered plaintext password, `computer`, for the `IETNova` account. No exploit, malicious code, or zero-day vulnerability was used. The recovery was the consequence of three preconditions: physical access to the disk, an unsalted hash format, and a weak human-chosen password.

![Ophcrack showing the recovered plaintext password for the IETNova account.](/assets/img/blog/ophcrack-rainbow-tables/image4.png)

## Key Observations

- Physical access to a non-encrypted disk effectively defeats SAM-based password protection. The lock on the SAM file is enforced by the running Windows kernel, not by the file system; booting an alternative OS removes that enforcement.
- Password length intuition that applies to modern hashing algorithms does not apply to LM. A 14-character LM-protected password is, in cryptographic terms, two independent 7-character secrets.
- Rainbow tables are effective against LM and NTLM specifically because both lack per-user salting. A salt of even modest length renders the precomputation approach infeasible.
- Ophcrack's free tables define what an attacker can recover with no investment. Passwords that survive those tables raise the attacker's cost, to paid tables, to GPU-accelerated tools such as `hashcat`, or to a different target altogether.

---

## Why This Attack Fails on Modern Windows

A reproduction of this exercise on a current Windows 11 system with default settings would not succeed. The defensive layers that have accumulated since Windows XP include:

- **LM hashing is disabled by default** beginning with Windows Vista. Modern systems store NTLMv2 only, eliminating the catastrophic 7-character split.
- **BitLocker** is enabled by default on supported hardware. Without the recovery key or a successful TPM unseal, the disk is opaque to a LiveCD environment.
- **Secure Boot** restricts which bootloaders the firmware will execute. Older Ophcrack live media generally do not satisfy Secure Boot signature requirements.
- **Credential Guard** (on Windows 11 Pro and Enterprise editions with virtualization-based security enabled) isolates LSASS secrets in a hypervisor-protected enclave, hardening the running system against in-memory hash extraction.
- **Microsoft Account-backed sign-in** shifts the relevant authentication secret server-side, so the local SAM is no longer the primary credential store on consumer devices.

The specific tooling demonstrated here is largely of historical interest. The underlying threat model, offline access to weakly hashed credentials, remains current. Equivalent scenarios include stolen laptops without full-disk encryption, leaked database dumps containing unsalted hashes, and improperly handled backup media.

---

## Defensive Recommendations

- Enforce full-disk encryption on all endpoints. This single control breaks the attack class demonstrated above.
- Disable LM hash storage on any legacy systems still emitting it (`NoLMHash` policy).
- Mandate password length and complexity sufficient to fall outside common precomputed tables, or move to passphrase-based authentication.
- Where possible, transition to passwordless authentication such as Windows Hello, FIDO2 security keys, or smart cards. Hashes that do not exist cannot be recovered.
- Treat any device that has been outside organizational control as potentially compromised, and rotate credentials accordingly.

---

## Conclusion

The Ophcrack exercise demonstrates that classical Windows password recovery succeeds when three conditions hold simultaneously: physical access to an unencrypted disk, a hash format without per-user salting, and a password within the coverage of available rainbow tables. Modern Windows breaks one or more of these conditions by default, but equivalent threat models persist whenever stored credentials are exposed offline. Subsequent posts will compare this rainbow-table approach against `hashcat` GPU-based attacks on the same hash formats to quantify the trade-off between precomputation storage and on-the-fly compute.
